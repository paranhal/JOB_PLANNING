## 1. 프로젝트 파일 구조 제안

처음에는 **동작 가능한 최소 구조**부터 두고, 입력/저장/조회가 안정된 뒤 단계적으로 확장한다.

```
job_planning/                    # 프로젝트 루트
├── pyproject.toml               # 의존성·패키징 (또는 requirements.txt)
├── README.md
├── src/
│   └── job_planning/
│       ├── __init__.py
│       ├── main.py               # 진입점: GUI 실행 또는 CLI
│       ├── config.py             # 경로·DB 경로·기본 설정

│       ├── db/                   # 저장·스키마

│       │   ├── schema.py         # 테이블 생성 DDL / 모델 정의
│       │   ├── connection.py     # SQLite 연결·초기화
│       │   └── crud/             # 엔티티별 CRUD (1차는 journal 위주)

│       │

│       │

│           ├── __init__.py

│           │   └── ...
│           └── widgets/          # 공통 위젯 (날짜 선택, 고객 콤보 등)
│               ├── __init__.py
│               └── common.py
│

    └── dist/
```

**1차(최소) 범위에서 실제로 만드는 디렉터리/파일:**

- `src/job_planning/`: `main.py`, `config.py`, `db/`(schema, connection, crud/journal), `services/journal_service.py`, `ui/app.py`, `ui/frames/journal.py`
- `data/`: DB 파일 위치(프로그램이 생성)
- 루트: `pyproject.toml` 또는 `requirements.txt`

고객·담당자·장비·점검·AS·내보내기는 **2단계 이후** 해당 폴더/파일을 추가하는 식으로 확장.

## 2. Python 구현 방식 제안

### 2-1. GUI 프레임워크 비교

| 방식 | 장점 | 단점 | 실행 파일 배포 |
|------|------|------|----------------|
| **tkinter** | 표준 라이브러리, 의존성 없음, 가벼움 | UI가 다소 구식, 고급 위젯 직접 구현 | ✅ PyInstaller와 조합 용이 |
| **CustomTkinter** | tkinter 기반, 현대적 look, 학습 곡선 낮음 | 외부 패키지 필요 | ✅ 동일 |
| **PySide6 / PyQt6** | 풍부한 위젯, 크로스플랫폼 | 무거움, 라이선스·의존성 | ✅ 가능, 번들 크기 큼 |
| **Streamlit** | 빠른 프로토타입, 웹형 UI | “앱” 느낌, 로컬 실행 파일로 묶기 불편 | ⚠️ 서버 실행 형태에 가까움 |

### 2-2. 권장: **CustomTkinter** (1차) 또는 **tkinter** (의존성 최소)

- **CustomTkinter:**  
  - 단일 실행 파일로 배포 가능, UI 품질 좋음, 문서·예제 많음.  
  - `pip install customtkinter` 한 개만 추가하면 됨.
- **tkinter:**  
  - 외부 패키지 없이 “일지 입력·저장·조회”만 먼저 만든다면 적합.  
  - 나중에 CustomTkinter로 전환해도 구조(`ui/app.py`, `ui/frames/journal.py`)를 그대로 재사용 가능.

**선택 가이드:**

- “의존성 없이 최소만” → **tkinter**
- “처음부터 조금 더 보기 좋은 창” → **CustomTkinter**

### 2-3. 저장소

- **1차:** SQLite 한 파일(`data/job_planning.db`).  
  - 표준 라이브러리 `sqlite3`만 사용해도 되고, 나중에 SQLAlchemy로 바꿔도 됨.
- **패키징:** PyInstaller로 `--onefile` 또는 `--onedir` 선택. DB 경로는 “실행 파일 기준”이 아니라 **사용자 데이터 디렉터리**(예: `~/Library/Application Support/JobPlanning` 또는 `%APPDATA%\JobPlanning`)에 두는 편이 안전.

### 2-4. 패키징

- **도구:** PyInstaller.  
  - CustomTkinter 사용 시: `customtkinter`를 hidden-import에 포함.  
- **1차:** `python -m PyInstaller src/job_planning/main.py --name JobPlanning ...` 형태로 스펙 파일 만들어 두고, “실행만 되는지” 확인하는 수준이면 충분.

## 3. 업무일지 입력·저장·수정·검색 — 함수/클래스 구조

“입력 → 저장 → 조회 → 수정”이 한 번에 이어지도록 **계층을 나누고**, 일지(journal) 중심으로만 먼저 설계한다.

### 3-1. 계층 구분

```
[UI]  ui/app.py, ui/frames/journal.py
        ↓ 사용자 입력 수집, 목록 표시, 수정/삭제 버튼
[서비스]  services/journal_service.py
        ↓ 검증, created_at/updated_at 자동 설정, 트랜잭션
[저장소]  db/crud/journal.py  ←  db/connection.py, db/schema.py
        ↓ SQL 실행
[SQLite]  data/job_planning.db
```

- **UI:** “무엇을 입력받고, 어떤 조건으로 검색할지”만 결정. 비즈니스 규칙은 서비스에 두지 않는다.
- **서비스:** “필수 필드 검증”, “날짜 형식”, “자동 필드(created_at, updated_at)” 처리. 필요 시 트랜잭션.
- **저장소(CRUD):** “테이블에 넣기/읽기/수정/삭제”만 담당. SQL/파라미터 바인딩만 다룬다.

### 3-2. DB 계층

**`db/connection.py`**

- `get_connection() -> sqlite3.Connection`  
  - 설정(`config.DB_PATH`)에 따라 DB 파일 열기, 없으면 생성.
- `init_db(conn)`  
  - `schema.py`의 DDL 호출해 테이블 생성(일지·고객·담당자·점검·AS·장비는 단계별로 추가).

**`db/schema.py`**

- `create_tables(conn)`  
  - `journal` 테이블 필수.  
  - 1차: journal만. 2단계에서 customer, assignee, equipment, 4단계 inspection_schedule, 5단계 as_request 추가.

**`db/crud/journal.py`**

- **create(conn, row: dict) -> id**  
  - `row`: date, customer_id(또는 null), work_type, content, assignee_id(또는 null), status, remarks.  
  - `created_at`, `updated_at`은 호출 전 서비스에서 넣거나, 여기서 `datetime.utcnow()`로 설정.
- **get_by_id(conn, id) -> dict | None**
- **list_(conn, date_from=None, date_to=None, customer_id=None, work_type=None, limit=500) -> list[dict]**  
  - 기간·고객·작업구분 필터, `ORDER BY date DESC`.
- **update(conn, id, row: dict) -> bool**  
  - `row`에 `updated_at` 반영.
- **delete(conn, id) -> bool**

행은 모두 **dict**로 주고받아서, UI·서비스에서 공통으로 사용하기 쉽게 한다.

### 3-3. 서비스 계층

**`services/journal_service.py`**

- **create(data: dict) -> id**  
  - `data`: date, work_type, content, (선택) customer_id, assignee_id, status, remarks.  
  - 검증: date/work_type/content 필수.  
  - `created_at`, `updated_at` 설정 후 `crud.journal.create` 호출.
- **get(id) -> dict | None**  
  - `crud.journal.get_by_id` 그대로 위임.
- **list_(date_from=None, date_to=None, customer_id=None, work_type=None)**  
  - `crud.journal.list_` 위임.
- **update(id, data: dict) -> bool**  
  - `updated_at` 갱신 후 `crud.journal.update`.
- **delete(id) -> bool**  
  - `crud.journal.delete` 위임.

검색은 “기간 + (선택) 고객 + 작업구분”으로 한정해 1차를 단순하게 둔다. “내용 키워드 검색”은 2차에서 추가해도 됨.

### 3-4. UI 계층 (일지만 1차)

**`ui/app.py`**

- 메인 창 생성.
- 탭 또는 메뉴: “업무일지”(1차만 구현), “고객”, “점검”, “AS” 등은 2단계 이후.
- “업무일지” 탭에 `JournalFrame` 배치.

**`ui/frames/journal.py` — JournalFrame**

- **입력 폼:**  
  날짜(date), 방문처/고객(1차는 텍스트 또는 나중에 콤보), 작업 구분(점검/AS/영업/기타), 작업 내용, 비고.  
  [저장] 클릭 → 서비스 `create()` 호출 → 성공 시 폼 초기화 및 목록 갱신.
- **목록(조회):**  
  기간(시작일~종료일) 선택 후 [조회] → 서비스 `list_()` → 테이블/리스트에 표시.  
  정렬: 날짜 역순.
- **수정/삭제:**  
  목록에서 행 선택 → [수정] 시 폼에 채워 넣고 저장 시 `update(id)` / [삭제] 시 `delete(id)` 후 목록 갱신.

1차에서는 “동작 가능한 최소”만: 입력 필드 + 저장 + 기간 조회 + 목록 + 수정·삭제.

### 3-5. 클래스/함수 요약표

| 계층 | 구성요소 | 역할 |
|------|----------|------|
| **db/connection** | `get_connection()`, `init_db(conn)` | DB 파일 연결, 테이블 생성 |
| **db/schema** | `create_tables(conn)` | journal 등 테이블 DDL |
| **db/crud/journal** | `create`, `get_by_id`, `list_`, `update`, `delete` | 일지 테이블 CRUD |
| **services/journal_service** | `create`, `get`, `list_`, `update`, `delete` | 검증·자동 필드·CRUD 오케스트레이션 |
| **ui/frames/journal** | `JournalFrame` (폼, 목록, 조회 버튼, 수정/삭제) | 일지 입력·저장·조회·수정·삭제 화면 |

고객·담당자·점검·AS·장비는 같은 패턴으로 `db/crud/xxx`, `services/xxx_service.py`, `ui/frames/xxx.py`를 단계별로 추가하면 된다.

## 4. 실행 파일 배포를 위한 기본 구조

- **진입점:** `main.py`에서 GUI만 실행하거나, `if __name__ == "__main__":` 에서 `ui.app.run()` 같은 한 줄로 창 띄우기.
- **경로:**  
  - 개발: `config.py`에서 `DB_PATH = 프로젝트/data/job_planning.db` 등 상대/절대 경로.  
  - 배포: PyInstaller로 묶을 때는 “실행 파일 위치”가 바뀌므로, **사용자 데이터 디렉터리**를 쓴다.  
    - macOS: `~/Library/Application Support/JobPlanning/job_planning.db`  
    - Windows: `%APPDATA%\JobPlanning\job_planning.db`  
  - `config.py`에서 `sys.frozen`(PyInstaller 설정 시) 여부로 분기해, 배포 시에는 위 경로를 쓰도록 하면 된다.
- **리소스:**  
  - 1차에는 아이콘·별도 리소스 없이 진행해도 됨.  
  - 나중에 아이콘 넣을 때는 `sys._MEIPASS`(PyInstaller 번들 내 리소스 경로)를 사용하는 패턴을 쓰면 된다.
- **의존성:**  
  - `requirements.txt` 또는 `pyproject.toml`에 `customtkinter`(또는 없으면 tkinter만), `openpyxl`(엑셀 내보내기 단계) 명시.  
  - PyInstaller 스펙에서 hidden-import에 `customtkinter` 등 필요한 패키지 넣기.

이렇게 해두면 “지금은 `python main.py`로만 실행하고, 나중에 PyInstaller로 한 번 묶어서 배포”하는 흐름이 자연스럽다.

## 5. 1차 구현 순서 — 바로 코딩 가능한 체크리스트

아래 순서대로 하면 “일지 입력 → 저장 → 기간 조회 → 수정·삭제”까지 동작하는 최소 구조를 만들 수 있다.  
코드는 **한 번에 전부 쓰지 말고**, 각 항목을 하나씩 구현·확인한 뒤 다음으로 넘긴다.

### Phase 0: 프로젝트 루트

- [ ] **0-1** 프로젝트 폴더 생성: `job_planning/` (또는 기존 JOB_PLANNING 아래 `job_planning/` 패키지)
- [ ] **0-2** `requirements.txt` 작성: `customtkinter` (또는 비움 — tkinter만 사용 시)
- [ ] **0-3** 가상환경 생성 후 `pip install -r requirements.txt`

### Phase 1: 설정·DB 기반

- [ ] **1-1** `src/job_planning/config.py`: `DB_PATH`, `DATA_DIR` 정의 (로컬 `data/` 또는 절대 경로)
- [ ] **1-2** `db/schema.py`: `journal` 테이블만 `create_tables(conn)`로 생성 (id, date, customer_id, work_type, content, assignee_id, status, remarks, created_at, updated_at)
- [ ] **1-3** `db/connection.py`: `get_connection()`, `init_db(conn)` 구현 (경로 없으면 디렉터리 생성)
- [ ] **1-4** 스크립트 또는 `main`에서 `init_db(get_connection())` 한 번 실행해 DB 파일·테이블 생성 확인

### Phase 2: 일지 CRUD

- [ ] **2-1** `db/crud/journal.py`: `create(conn, row)` 구현 (row에 created_at, updated_at 포함)
- [ ] **2-2** `db/crud/journal.py`: `list_(conn, date_from, date_to, ...)` 구현
- [ ] **2-3** `db/crud/journal.py`: `get_by_id`, `update`, `delete` 구현
- [ ] **2-4** 콘솔 또는 간단 스크립트로 create → list_ → get_by_id → update → delete 동작 확인

### Phase 3: 서비스

- [ ] **3-1** `services/journal_service.py`: `create(data)` — 필수 필드 검증, created_at/updated_at 설정 후 crud.create 호출
- [ ] **3-2** `get(id)`, `list_(...)`, `update(id, data)`, `delete(id)` 위임 구현
- [ ] **3-3** (선택) 콘솔에서 서비스 create/list_ 호출해 동작 확인

### Phase 4: GUI — 일지 화면

- [ ] **4-1** `main.py`: CustomTkinter 또는 tkinter로 메인 창 띄우기, `JournalFrame` 하나만 배치
- [ ] **4-2** `ui/frames/journal.py`: 입력 폼 (날짜, 작업구분, 내용, 비고) + [저장] 버튼 → journal_service.create 호출
- [ ] **4-3** 기간(시작일, 종료일) 선택 + [조회] 버튼 → journal_service.list_ 호출 → 결과를 테이블/Listbox에 표시
- [ ] **4-4** 목록에서 행 선택 시 [수정] → 폼에 채움 → [저장] 시 update 호출
- [ ] **4-5** 목록에서 행 선택 시 [삭제] → delete 호출 후 목록 갱신
- [ ] **4-6** 저장/수정/삭제 후 “성공” 메시지 또는 폼 초기화로 UX 정리

### Phase 5: 실행·정리

- [ ] **5-1** `python -m src.job_planning.main` 또는 `python main.py`로 전체 실행 확인
- [ ] **5-2** DB 파일이 의도한 경로에 생성·유지되는지 확인
- [ ] **5-3** (선택) PyInstaller로 `main.py` 단일 실행 파일 생성해 실행만 되는지 확인

## 6. 정리

- **파일 구조:** `src/job_planning` 아래에 `config`, `db`, `services`, `ui`로 계층 분리. 1차는 일지 관련만 구현.
- **구현 방식:** GUI는 CustomTkinter(권장) 또는 tkinter, 저장은 SQLite, 패키징은 PyInstaller.
- **입력/저장/수정/검색:** DB(connection, schema, crud) → 서비스(검증·자동 필드) → UI(폼·목록·버튼) 순으로 설계했으며, 일지만 먼저 완성 후 고객·담당자·점검·AS·장비·내보내기를 단계별로 붙이면 된다.
- **실행 파일:** config에서 사용자 데이터 디렉터리 사용, PyInstaller 스펙만 준비해 두면 이후 배포 시 확장 가능.
- **1차 체크리스트:** Phase 0~5를 순서대로 진행하면 “일지 입력 → 저장 → 기간 조회 → 수정·삭제”가 안정적으로 동작하는 최소 프로그램을 만들 수 있다.

이 문서를 구현 탭의 설계 기준으로 사용하고, 각 Phase를 완료할 때마다 체크리스트를 갱신하면 된다.
