## 0. 기획 기능 및 구현 방향 요약

### 0-1. 기획 에이전트 규칙에서 정의한 기능(agent-planning.mdc)

| 영역 | 기획 기능 요약 | 구현 시 반영 방향 |
|------|----------------|-------------------|
| **고객 정보** | 고객번호, 고객명, 정식명칭(Full Name), 지역, 주소, 전경 사진, 층·실 수, 업무 담당자 | 통합 고객 마스터: 필드 확장 가능 구조로 설계. 1차는 고객명·구분·연락처 등 최소, 2차에서 정식명칭·주소·사진·층실 수·담당자 연결 확장 |
| **담당자 정보** | 담당자번호, 담당자명, 연락처, 핸드폰, 이메일, 개인 이메일, 소속, 소속일자, 근무장소, 담당업무 | 통합 담당자(contact): 고객별 1:N. 1차는 담당자명·직장/핸드폰·이메일, 2차에서 소속·소속일자·근무장소·담당업무·개인이메일 확장 |
| **장비 정보** | 장비고유번호, 장비분류(RFID/KLAS/기타), 장비형태(SW/HW), 장비명, 제조사, 모델명, 시리얼번호, 제조일자, 사내 담당자, 담당등급(정/부), 배정일자, 유지보수 등급(무상/유상/미계약/기타) | 통합 장비 마스터: 고객별 1:N. 1차는 장비명·설치위치·점검주기·유무상, 2차에서 분류·형태·제조사·모델·시리얼·담당자·등급·배정일자 확장 |
| **설치 정보** | 장비고유번호, 설치장소, OS 버전, IP, 서브넷, 게이트웨이, 팀뷰어 ID/패스워드, 설치일자 | 장비와 1:1 또는 장비 테이블 확장 컬럼. 2차에서 설치정보 엔티티 또는 장비 확장 필드로 구현 |
| **유지보수 정보** | 장비고유번호, 접수일자, 접수 담당자, 접수 방법(유선/대면/원콜센터/제조사/기타), 접수내용, 접수분류(고장수리/설정변경/기능상담/기타), 조치일, 조치내용, 조치분류(조치완료/재방문필요/이관/유선처리/기타), 종료여부(Y/N), 완료일 | **통합 처리 이력(work_log)** 한 종류로 관리. 구분(제조사·품목·지역) + 접수방법·접수내용·조치일·조치내용·조치분류·종료여부·완료일 등 필드로 확장. 기획서 §2-1과 통합 |

### 0-2. 기획 에이전트 규칙에서 정의한 업무 흐름 → 구현 방안

| 업무 흐름 | 기획 내용 | 구현 방안 |
|-----------|-----------|------------|
| **1. 접수 및 배당** | 유지보수 접수 시 접수대장 기록, 지정 사내 담당자에게 배당·전달, 부재 시 전체 전파. 접수 방법: 유선/대면/원콜센터/제조사/기타 | 통합 처리 이력에 “접수일자·접수방법·접수 담당자(고객/사내)” 저장. 1차: 처리 이력 입력 시 담당자(contact) 선택. 2차: 배당·알림(전체 전파) UI/로직 |
| **2. 조치** | 조치 일시·조치 담당자(자동/수동), 조치 시작 버튼 시 자동 입력. 조치 내용·조치 분류(조치완료/재방문필요/이관/유선처리/기타). 고객 담당자 사인(유선처리 시 제외) | 처리 이력에 조치일·조치내용·조치분류·종료여부·완료일 필드. 1차: 수동 입력. 2차: “조치 시작” 시 조치일시·담당자 자동 기록, 사인(서명) 필드 또는 별도 플래그 |
| **3. 접수대장** | 접수일자, 접수방법, 접수 담당자(고객/사내), 조치일시, 조치내용, 현재 상태(진행중/조치완료). 담당자별·고객별 현황 | 통합 처리 이력 목록 = 접수대장. 필터: 기간·접수방법·구분·담당자·고객·현재 상태(진행중/완료). 2차: 담당자별/고객별 요약 뷰 또는 통계 탭 |
| **4. 통계** | 장비분류별 처리 완료 기간, 담당자별 통계, 고객별 통계, 장비분류별 통계 | 2차 기능. 통합 처리 이력·고객·장비 데이터 기반 집계 화면 또는 내보내기 |

### 0-3. 기획서(요구사항)와의 정합성

- **통합 처리 이력:** 구분(확장 가능)·접수 방법(유선/대면/메일/기타/원콜센터)·기간·고객·진행 상태로 입력·조회·수정·삭제. 고객/담당자/장비는 통합 마스터에서 선택.
- **통합 고객·담당자·장비:** 한 마스터에서 CRUD, 구분 필터, 신규 구분 추가 가능.
- **통합 월별 일정:** 연·월별 달력, 날짜별 방문처·고객 목록·메모.
- **저장:** 로컬(SQLite 또는 JSON) 즉시 반영.
- **내보내기:** 통합 데이터 → 엑셀(2차에서 기존 시트 양식 매핑).

## 2. Python 구현 방식 제안

### 2-1. GUI

- **현행:** PySide6(Qt) 사용 시 그대로 유지. 풍부한 위젯·크로스플랫폼.
- **대안:** CustomTkinter / tkinter — 의존성 최소화 시 선택. 구조(ui/app.py, frames) 동일하게 재사용 가능.

### 2-2. 저장소

- **현행:** JSON 파일(`store/_json_store.py` + `data/*.json`). 로컬 단일 사용자에 적합.
- **대안:** SQLite(`db/schema.py`, `db/connection.py`, `db/crud/*`) — 기획서 §2-4·§5 반영 시. 배포 시 사용자 데이터 디렉터리 사용 권장.

### 2-3. 패키징

- PyInstaller로 단일 실행 파일 또는 onedir. DB/JSON 경로는 실행 파일 기준이 아닌 사용자 데이터 디렉터리 권장.

## 3. 통합 처리 이력(work_log) — 입력·저장·조회·수정

### 3-1. 계층 구분

```
[UI]  ui/frames/work_log.py  (폼·목록·필터·수정/삭제)
        ↓
[서비스]  services/work_log_service.py  (검증, 구분·접수방법·상태 검사, 자동 필드)
        ↓
[저장소]  store/work_log.py  (JSON 또는 db/crud/work_log.py)
        ↓
[데이터]  data/work_log.json (또는 SQLite)
```

### 3-2. 통합 처리 이력 데이터 항목(기획 반영)

| 항목 | 필수/선택 | 비고 |
|------|-----------|------|
| **구분(source)** | 필수 | 확장 가능. 예: NICOM, 원콜, 세종KLAS, 당사 홈페이지, 기타. config 또는 코드 테이블에서 관리 |
| **일시(occurred_at)** | 필수 | 접수·발생 기준 |
| **진행 상태(status)** | 필수 | 진행중, 완료, 처리완료, 보류 |
| **고객(customer_id)** | 권장 | 통합 고객 마스터 연결 |
| **담당자(contact_id)** | 선택 | 고객별 담당자 마스터 연결 |
| **접수 방법** | 선택 | 유선, 대면, 메일, 기타, 원콜센터 (기획 에이전트 규칙·기획서 §2-1) |
| **접수 내용(received_content)** | 선택 | NICOM형 등 |
| **처리/조치 내용(handled_content)** | 선택 | 조치일(processed_at), 조치분류(2차 확장) |
| **분류(category)** | 선택 | 메일문의, 전화문의, 홈페이지, 정기방문, 현장방문, 기타 |
| **제목·내용·답변** | 선택 | 원콜/세종KLAS형 |
| **모델명·장비위치·유무상** | 선택 | NICOM형 |
| **비고(remarks)** | 선택 | 공통 |
| **created_at, updated_at** | 자동 | 서비스/저장소에서 설정 |

2차에서 **조치분류(조치완료/재방문필요/이관/유선처리/기타), 종료여부, 완료일** 등 기획 규칙의 유지보수 정보 필드 확장.

### 3-3. 서비스 계층 (work_log_service)

- **create(data)** — 구분·일시·진행·내용(또는 접수/처리 내용) 등 필수 검증; 구분·접수방법·상태는 config 허용값 검사; created_at/updated_at 설정 후 store.create.
- **get(id)**, **list_(source=, date_from=, date_to=, customer_id=, status=, 접수방법=)** — store 위임.
- **update(id, data)** — 동일 검증 후 store.update.
- **delete(id)** — store.delete.

### 3-4. UI 계층 (WorkLogFrame)

- **입력 폼:** 구분(콤보), 일시, 진행, 고객(콤보)·담당자(콤보), 접수방법, 내용/접수내용/처리내용, 구분별 추가 필드(NICOM: 모델·장비위치·유무상; 원콜/세종KLAS: 처리일자·분류·제목·답변), 비고. [저장] / [새로 작성].
- **목록(접수대장):** 필터(구분·시작일·종료일·접수방법·고객·상태) → [조회] → work_log_service.list_ → 테이블 표시. [수정] / [삭제].
- 고객 선택 시 담당자 콤보 갱신(contact_service.list_by_customer).

## 4. 통합 고객·담당자·장비 마스터

### 4-1. 고객(customer)

- **1차 필드:** id, 고객명(name), 구분(division), 연락처/비고(phone). (기획 규칙의 고객번호·정식명칭·지역·주소·전경 사진·층실 수·업무 담당자는 2차 확장)
- **store/customer.py**, **services/customer_service.py**, **ui/frames/master.py** 내 CustomerTab — CRUD, 구분 필터.

### 4-2. 담당자(contact)

- **1차 필드:** id, customer_id, 담당자명(name), 직장/연락처(phone_office), 핸드폰(phone_mobile), 이메일(email). (기획 규칙의 소속·소속일자·근무장소·담당업무·개인이메일은 2차)
- **store/contact.py**, **services/contact_service.py**, MasterFrame 내 ContactTab — 고객별 목록·CRUD.

### 4-3. 장비(equipment)

- **1차 필드:** id, customer_id, 장비명(name), 설치위치(location), 점검주기(inspection_interval), 유무상(billing_type). (기획 규칙의 장비분류·형태·제조사·모델·시리얼·담당자·등급·배정일자, 설치정보는 2차)
- **store/equipment.py**, **services/equipment_service.py**, MasterFrame 내 EquipmentTab — 고객별 목록·CRUD.

### 4-4. 구분(division/source) 확장

- config 또는 별도 JSON/테이블에 구분 목록 보관. 관리자/설정에서 신규 구분 추가 가능하도록 설계(2차에서 UI 제공).

## 5. 통합 월별 일정(2차 포함)

- **store:** inspection_schedule 또는 calendar 전용 저장소.

## 7. 정리

- **기준:** 기획 에이전트 규칙(agent-planning.mdc)의 고객지원 관리 기능·업무 흐름과, 기획_업무일지_프로그램_요구사항.md의 통합 구조·MVP·개발 순서를 함께 반영한다.
- **통합 처리 이력:** 한 종류의 이력으로 접수·조치·접수대장·(2차)통계를 지원하며, 구분·접수방법은 확장 가능하게 둔다.
- **통합 고객·담당자·장비:** 1차는 최소 필드로 CRUD, 2차에서 기획 규칙의 상세 필드(정식명칭·주소·장비분류·설치정보·유지보수 등급 등) 및 업무 흐름(배당·조치 시작·사인)을 확장한다.
- **구현:** UI → 서비스(검증·자동 필드) → 저장소(JSON 또는 SQLite) 3계층 유지. 현재 코드 구조(store/services/ui)와 정합성을 맞추고, 문서는 기획 기능·업무 흐름 확장에 대비한 확장 지점을 명시한다.

이 문서를 구현 탭의 설계 기준으로 사용하고, Phase 완료 시 체크리스트를 갱신한다.
