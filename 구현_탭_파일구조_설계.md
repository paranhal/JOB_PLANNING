# 구현 탭 — 파일구조·UI 설계

**기준 문서:** `기획_업무일지_프로그램_요구사항.md` (고객지원 시스템 구축 기획서)  
**데이터 설계:** `데이터_처리_설계_PostgreSQL확장.md`  
**화면 참고:** `화면설계서.pptx` / `화면설계서_요약.md` (있을 경우)

본 문서는 **기획서 §13 화면(메뉴) 구성안**과 **§16 단계별 구축 권장안**을 반영하여, 메뉴·탭·프로젝트 파일 구조를 정리한다.

---

## 1. 메뉴·탭 구조 (§13 반영)

기획서 §13 화면(메뉴) 구성안을 그대로 상위 메뉴로 두고, **탭** 또는 **서브메뉴**로 화면을 구성한다.

### 1.1 전체 메뉴 맵

| 대메뉴 | 화면(탭) | 핵심 기능 | 구축 단계 |
|--------|----------|-----------|-----------|
| **기준정보** | 고객관리 | 기관 기본정보, 상위기관, 주소, 연락처 | 1단계 |
| **기준정보** | 공간관리 | 건물/층/실 등록 및 조회 | 1단계 |
| **기준정보** | 담당자관리 | 현재 담당자 등록, 주담당 지정 | 1단계 |
| **기준정보** | 담당자이력 | 기관별/담당자별 변경 이력 조회 | 1단계 |
| **자산관리** | 설치자산관리 | 제품, 위치, 설치주체, 관리유형 등록 | 1단계 |
| **자산관리** | SW상세관리 | OS, 버전, DBMS, 접속방식 등록 | 1단계 |
| **자산관리** | 수행관계관리 | 제조사/협력사/원청 관계 관리 | 3단계 |
| **AS관리** | AS접수 | 접수 등록, 배정, 긴급도 설정 | 2단계 |
| **AS관리** | AS처리 | 처리내역, 원인, 조치결과 입력 | 2단계 |
| **AS관리** | AS현황 | 미처리, 지연, 기관별, 자산별 현황 | 2단계 |
| **분석/영업** | 교체대상현황 | 노후장비, 장애다발장비, 타사설치 자산 분석 | 3단계 |
| **관리** | 코드관리 | 표준값(드롭다운) 관리 | 1단계 |
| **관리** | 권한관리 | 사용자 권한 관리 | 후속 |

### 1.2 1단계에서 구현할 탭 (§16 1단계)

- **기준정보** 탭 그룹: 고객관리 · 공간관리 · 담당자관리 · 담당자이력
- **자산관리** 탭 그룹: 설치자산관리 · SW상세관리 (설치자산 상세 내 서브 또는 별도 탭)
- **관리** 탭 그룹: 코드관리

UI 구성 예: **좌측 트리/리스트 = 대메뉴**, **상단 또는 좌측 2단 = 화면(탭)** 선택 후 **우측 = 목록·상세·입력 폼**.

## 2. 탭별 화면 요약 (1단계)

| 탭 | 목록 영역 | 상세/입력 영역 | 비고 |
|----|-----------|----------------|------|
| 고객관리 | 기관 목록(기관명, 공식명칭, 업종, 사용여부) | 기관 기본정보 폼(§5.1), 상위기관 선택 | 검색: 기관명, 공식명칭 |
| 공간관리 | 건물 목록 → 선택 시 층/실 트리 또는 목록 | 건물/층/실 등록·수정 폼(§5.2) | 건물→층→실 계층 |
| 담당자관리 | 기관별 담당자 목록 | 담당자 등록·수정(§5.3), 주담당 지정 | 재직상태, 부임/퇴임일 |
| 담당자이력 | 기관별/담당자별 필터 후 이력 목록 | 이력 조회(읽기 위주)(§5.4) | 시작일/종료일, 변경사유 |
| 설치자산관리 | 기관별/제품별 설치 목록 | 설치기본정보 폼(§6.1), 위치=건물·층·실 선택 | 제품구분, 설치주체, 관리유형 코드화 |
| SW상세관리 | 설치자산 선택 후 해당 SW 상세 목록 | SW상세 폼(§6.2) | 설치ID와 1:N 가능 |
| 코드관리 | 코드그룹 선택 → 코드 값 목록 | 코드 값 추가/수정/정렬, 사용여부 | §11 코드 그룹 전부 |

## 3. 프로젝트 파일 구조

1단계 기준 **Python + 데스크톱 GUI**(예: PyQt6 / Tkinter) + **JSON 또는 SQLite** 저장을 전제로 한 구조이다. 데이터 계층 필드명은 `데이터_처리_설계_PostgreSQL확장.md`와 동일하게 유지한다.

- **2단계**에서 AS·첨부 추가 시 `store`/`services`/`frames`에 as_reception, as_action, attachment 반영.
- **3단계**에서 수행관계·교체대상현황 추가 시 `relation`, `replacement` 프레임 및 서비스 확장.

## 4. 메인 윈도우·탭 배치 예시

- **좌측:** 대메뉴(기준정보 | 자산관리 | AS관리 | 분석/영업 | 관리). 1단계에는 기준정보·자산관리·관리만 활성화.
- **중앙 상단 또는 좌측 2단:** 선택된 대메뉴 하위 탭(예: 기준정보 → 고객관리 | 공간관리 | 담당자관리 | 담당자이력).
- **중앙:** 선택된 탭의 목록 + 상세/입력 영역(목록 좌측·상세 우측, 또는 상단 목록·하단 상세).
- **검색:** 기획서 §15에 따라 기관명, 공식명칭, 담당자명, 제품명, S/N 등 통합 검색 가능한 공통 위젯(`entity_search`) 사용 권장.

## 5. 기술 스택 제안

| 구분 | 1단계 권장 | 비고 |
|------|------------|------|
| 언어 | Python 3.10+ | |
| GUI | PyQt6 또는 tkinter | 데스크톱 단일 사용자 |
| 저장소 | JSON 파일 또는 SQLite | `데이터_처리_설계_PostgreSQL확장.md` 필드명 준수 |
| 패키징 | pyproject.toml + venv | 의존성 고정 |

PostgreSQL 전환 시에는 `store` 계층만 교체(또는 어댑터 추가)하고, `services`·`ui`는 필드명을 그대로 쓰면 된다.

## 6. 1단계 구현 순서 제안

1. **config + store 공통**  
   `config.py`, `store/_base.py`, 데이터 디렉터리 생성. 코드관리(`code_master`) 저장·로드.
2. **코드관리 UI**  
   코드 그룹/값 CRUD. 다른 화면에서 드롭다운 바인딩용.
3. **고객 마스터**  
   store·service·고객관리 프레임. 상위기관 선택 시 코드/고객 목록 참조.
4. **공간(건물·층·실)**  
   store·service·공간관리 프레임. 계층 선택 후 등록/수정.
5. **담당자·담당자이력**  
   store·service·담당자관리·담당자이력 프레임. 기관별 필터, 이력은 조회 위주.
6. **설치자산·SW상세**  
   store·service·설치자산관리·SW상세 프레임. 위치=건물/층/실 선택, 코드 연동.
7. **메인 앱 통합**  
   `app.py`에서 대메뉴·탭 연결, 1단계 메뉴만 노출.

이 순서로 하면 코드관리·고객·공간·담당자·설치자산까지 **§13·§16 1단계**의 메뉴·탭·파일 구조가 정리된 상태로 구현할 수 있다.
