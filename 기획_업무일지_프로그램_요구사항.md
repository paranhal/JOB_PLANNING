## 1. 추진 배경 및 목표

본 시스템은 고객기관의 기본 정보와 담당자, 설치자산, 수행관계, 지원이력을 표준화하여 AS 접수·처리·현황 관리를 체계화하기 위한 고객지원 시스템이다.

특히 고객기관 내 설치 장비의 상당수가 타사 납품 또는 과거 설치 자산인 현실을 반영하여, 당사 계약 유무와 무관하게 기관 전체 설치환경을 관리하고 향후 교체 시점에 영업 기회를 발굴할 수 있도록 설계한다.

### 1.1 추진 배경

- 고객기관의 장비·시스템은 다수의 제조사, 협력사, 과거 설치업체가 혼재되어 있어 현장 정보가 분산됨.
- 담당자 교체가 잦아 현재 담당자만 관리하면 업무 이력과 대응 맥락이 단절됨.
- AS 접수 시 설치 위치, 제품 정보, 설치환경, 요청 주체를 즉시 확인할 수 있는 기준정보가 필요함.
- 타사 설치 장비라도 교체주기, 노후도, 장애 빈도를 파악하면 향후 교체 영업과 제안 활동으로 연결 가능함.

### 1.2 시스템 목표

- 고객기관/공간/담당자/설치자산 정보를 일관된 구조로 축적한다.
- AS 접수부터 처리 완료, 후속조치까지 전 과정을 이력으로 남긴다.
- 현재 운영 자산과 과거 이력을 함께 관리하여 장애 대응 정확도를 높인다.
- 노후장비, 장애다발장비, 지원빈도 높은 기관을 식별하여 교체 및 업그레이드 영업에 활용한다.

## 2. 시스템 성격과 기본 원칙

### 2.1 시스템 정의

본 시스템은 일반적인 계약관리형 CRM이 아니라, 기관 중심의 운영형 고객지원 시스템이다. 핵심 관리 축은 기관 기준정보, 공간정보, 담당자 및 이력, 설치자산, 수행관계, AS 운영정보이다.

### 2.2 설계 원칙

- 계약 여부와 무관하게 고객기관 내 운영대상 자산을 관리한다.
- 모든 주요 엔터티는 이름이 아닌 고유 ID로 연결한다.
- 현재 정보와 이력 정보를 분리하여 변경 추적이 가능하도록 한다.
- 설치 위치는 자유입력 최소화, 건물-층-실 구조 기반으로 표준화한다.
- AS 정보는 설치자산과 연결하여 동일 장비의 반복 장애와 교체주기를 추적한다.
- 비밀번호 등 민감정보는 평문 저장을 지양하고 암호화 파일 또는 별도 금고 체계와 연결한다.

### 5.4 담당자 이력 관리

- 현재 담당자 정보는 최신값만 유지하고, 변경 시 과거 이력은 별도 행으로 보존한다.
- 기관 이동, 업무 변경, 연락처 변경, 퇴임/전보 등 모든 변동은 시작일/종료일 기준으로 관리한다.
- 특정 AS 건이 발생한 시점의 실제 담당자를 재현할 수 있어야 한다.

| 항목 | 설명 |

| 담당자이력ID | 이력 고유 식별자 |
| 담당자ID / 고객ID | 해당 담당자와 기관 연결 |
| 시작일 / 종료일 | 유효기간 |

### 6.3 접속정보 관리 원칙

- ID/PASSWORD는 시스템 DB에 평문 저장하지 않는다.
- 접속정보는 암호화 파일, 금고 시스템 또는 접근권한 제한 저장소에 보관한다.
- 시스템에는 접속정보 파일경로, 저장방식, 최종확인일, 관리주체만 저장한다.

## 8. AS 접수·처리·현황 관리 설계

### 단계별 처리

| 단계 | 핵심 처리 | 주요 입력/출력 |
|------|-----------|----------------|
| 1. 접수 | 기관·설치자산 식별, 증상 등록 | 접수번호, 고객, 설치ID, 요청내용, 긴급도 |
| 2. 분류 | 장애유형, 제품구분, 요청주체, 담당 배정 | 담당자, SLA, 처리우선순위 |
| 3. 처리 | 원인분석, 현장/원격조치, 부품교체, 협력사 연계 | 처리내역, 원인, 사용부품 |
| 4. 완료 | 조치결과 확인 및 완료처리 | 완료일시, 결과코드, 고객확인 |
| 5. 후속 | 재발방지, 교체검토, 영업전환 | 후속조치, 교체대상 여부, 영업메모 |

|------|------|------|
| AS번호 | 접수 건 고유 ID | 자동생성 |
| 접수일시 | 접수 시간 | 필수 |
| 고객ID / 설치ID | 기관 및 자산 식별 | 가급적 설치자산 연결 |
| 접수채널 | 전화, 이메일, 방문, 협력사 요청 등 | 코드화 |

## 14. 핵심 업무 규칙

- 기관명과 공식명칭은 분리 관리한다.
- 사업자번호 또는 내부 중복 검증 규칙으로 동일 기관 중복 등록을 방지한다.
- 담당자는 삭제하지 않고 재직상태와 이력으로 관리한다.
- 설치자산의 위치는 건물-층-실 기준으로 선택하고 상세위치만 보조 입력한다.
- 설치주체, 관리유형, 요청주체는 반드시 코드화한다.
- AS 접수는 가급적 설치자산과 연결하며, 연결이 불가능한 경우 임시 접수 후 자산 매핑한다.
- 처리 완료 시 후속조치와 교체검토 여부를 함께 기록한다.
- 노후도와 장애빈도 기준으로 교체영업 후보를 자동 또는 반자동 산출한다.
- 정기 점검 일정은 자동 초안 이후에도 담당자 수정이 가능하며, 승인 전에는 외부 캘린더에 반영하지 않는다.

- 교체대상 분석은 설치연수, 최근 1년 장애건수, 처리비용, 부품단종 여부를 결합해 확장 가능하다.

- 개인정보 및 접속정보는 접근권한, 로그기록, 암호화 저장 정책을 적용해야 한다.
- 정기 점검 모듈은 지도 API 의존도를 낮추기 위해 이동시간 **수동 대체 입력**과 **캐시**를 동시에 지원하는 것이 바람직하다.
- 엑셀 월별 캘린더(17.11)는 **병합 셀·스타일·시트 다중 생성**이 필요하므로, 서버에서 xlsx 생성 라이브러리(Go/ Python 등) 선정 시 **스타일 API 지원 여부**를 검증한다.

## 17. 정기 점검 일정 자동 생성

매월 반복되는 **정기 점검** 일정을 시스템이 초안으로 수립하고, 담당자가 검토·수정·승인한 뒤 개인/팀 캘린더와 연동하는 기능을 정의한다. 기존 **고객기관(사이트)·공간·설치자산** 정보와 연계하여, 사이트별 점검 대상 영역(KLAS / RFID 자동화 장비) 및 장비 수를 반영한다.

### 17.1 목적 및 성격

- 월간 정기 점검 대상이 많을 때 수작업 배열을 줄이고, **지역·이동시간·업무시간** 제약을 만족하는 일정 초안을 자동으로 만든다.
- 자동 생성 결과는 **초안**이며, 반드시 담당자가 현장 상황에 맞게 수정할 수 있어야 한다.
- **승인 완료**된 일정만 외부 캘린더(구글 캘린더 또는 애플 캘린더)에 반영한다. **엑셀 월별 캘린더**는 초안·승인본 모두 보낼 수 있으나, 대외 공유 시에는 승인본 사용을 권장한다.

### 17.2 점검 영역 및 사이트 단위 규칙

| 점검 영역 | 설명 |
|-----------|------|
| **KLAS** | 하드웨어 제품 단위 점검이 아닌 영역으로, **사이트당 표준 소요시간 20분**을 기본값으로 둔다. |
| **RFID 자동화 장비** | 장비(설치자산) 수에 비례한 점검으로, **장비당 표준 소요시간 약 15분**을 기본값으로 둔다. |

- 사이트(고객기관)마다 **KLAS만**, **RFID만**, **둘 다** 대상인 경우가 있다. 기준정보에서 사이트별로 대상 영역을 지정할 수 있어야 한다.
- 한 사이트 방문에서 두 영역을 연속 수행하는 것을 기본으로 하되, 일정 편집 시 분리·재배치 가능하도록 한다.

### 17.3 담당 직원 배정

- 월간 정기 점검에는 **전담 직원**을 미리 배정할 수 있다. 배정은 **필수가 아니다.**
- 배정이 없거나 변경이 필요한 경우, **업무 배정 수정** 기능으로 담당자를 바꿀 수 있어야 한다(자동 초안 이후에도 동일).

### 17.4 경로·지역·이동 시간

- **이동 경로 및 일자 배치** 시 **지역별로 묶어 계획하는 것을 최우선**으로 한다(동일 권역 내 방문 순서 최적화).
- 사이트 간 **이동 소요 시간**은 원칙적으로 **네이버 지도(또는 동등한 경로·거리 API)** 를 통해 산정한다.
- **구현이 불가능한 경우**(API 미계약, 정책 제한 등): 담당자가 **사이트 쌍 또는 지역 간 표준 이동시간**을 수동 입력·관리할 수 있는 대체 수단을 제공한다.

### 17.7 자동 수립 로직(요구 수준)

시스템은 최소 다음 순서를 만족하는 알고리즘(또는 이에 준하는 휴리스틱)을 적용한다.

1. 고정 규칙 사이트를 해당 월의 유효한 법정근무일에 배치한다.
2. 잔여 사이트를 **지역** 단위로 클러스터링한다.
3. 각 근무일에 대해 사무실 출발 시각(기본 09:30~10:00)부터, **이동시간 + 현장 소요시간**을 누적하여 배치하며, **17:00 전 마지막 도착·종료**를 넘지 않는 선에서 일자를 나눈다.
4. 동일 일자·동일 권역 내 방문 순서는 이동시간 합이 작아지도록 정렬·최적화한다(정확한 TSP 필수는 아니나, 무리한 왕복을 줄이는 수준 이상).
5. **25일 이전** 완료가 불가능하면 **경고·미배치 목록**을 생성하고 담당자가 수동으로 조정한다.

### 17.8 수정·승인·버전

- 자동 생성 직후 상태는 **초안**이다. 담당자는 **날짜·순서·담당자·현장 시간**을 수정할 수 있어야 한다.
- **승인** 절차(승인자·승인 시각)를 두어, 승인된 버전만 외부 연동·대외 공유의 기준이 되도록 한다.

### 17.9 캘린더 연동

- **승인된 일정**에 한하여 **구글 캘린더** 또는 **애플 캘린더**에 등록할 수 있어야 한다.
- 연동 방식은 구현 단계에서 확정하되, OAuth(구글), CalDAV/구독 URL(애플) 등 **표준적 수단**을 우선 검토한다. 개인 계정/공용 팀 캘린더 여부는 운영 정책으로 정한다.

### 17.10 데이터·기준정보 연계(구현 시)

- **사이트**: 고객마스터(기관)와 1:1 또는 명시적 매핑.
- **지역**: 주소·시도·권역 코드 등 기준정보 또는 별도 **권역 마스터**.
- **RFID 점검 시간**: 설치자산 중 해당 분류 장비 수 × 기본 15분(사이트별 오버라이드 가능).
- **KLAS**: 사이트당 20분 기본(오버라이드 가능).
- **이동시간**: API 결과 캐시 + 수동 보정 테이블.

### 17.11 엑셀 월별 캘린더 출력(유지보수계획표)

현업에서 사용하는 **유지보수계획 엑셀**(월별 달력 형태)과 **동등한 정보 구조·가독성**을 갖춘 파일을 시스템에서 생성·보내야 한다. 참고 레이아웃: 상단에 연·월 제목, 요일별 7열 격자, 날짜 칸에 사이트명 다줄 표기, **1월~12월 시트 분리**, 글자색으로 방문 유형 구분.

#### 17.11.1 파일·시트 구성

| 항목 | 요구 |
|------|------|
| 시트 | **12개**: 시트 이름 `1월` … `12월`(해당 **연도** 기준). 활성 시트는 사용자가 열 때 해당 월로 두어도 되고, 기본은 `1월` 또는 “오늘이 속한 월”로 둘 수 있다. |
| 파일명 | 예: `YYYYMMDD유지보수계획.xlsx`. 날짜·연도·버전 규칙은 운영 정책으로 정하되, **다운로드 시점 또는 계획 기준일**이 식별되도록 한다. |
| 인코딩·글꼴 | 한글 사이트명이 깨지지 않아야 하며, Windows Excel 기본 환경(예: **맑은 고딕**)에서 정상 표시된다. |

#### 17.11.2 달력 격자(각 월 시트)

- **열 머리글**: **일요일, 월요일, … 토요일**(한글 요일명, 왼쪽부터 일요일 시작).
- **월 표제**: 해당 시트 상단에 **연·월**을 크게 표시한다. 예: `2026년 3월`. 제목은 눈에 띄게(글꼴 크기·**녹색 등 강조색**) 구성할 수 있다.
- **날짜 칸(셀)**:
  - 칸 **상단 왼쪽**에 해당 일의 **일자(숫자)**.
  - 그 **아래**에 그날 방문 예정 **사이트 표시명**을 **줄바꿈**으로 나열한다. 한 날짜에 **복수 사이트**가 있으면 모두 표기한다.
- **해당 월에 속하지 않는 칸**(앞뒤 빈 칸)은 비우거나 배경만 구분해 달력 형태를 유지한다.

#### 17.11.3 비근무일·빈 칸

- **토요일·일요일 및 공휴일**(17.5와 동일 데이터 기준)에는 **점검 일정을 넣지 않는다**. 필요 시 해당 칸에 **회색 배경** 등으로 휴무임을 표시할 수 있다.
- 일정이 없는 평일 칸은 비워 둔다.

#### 17.11.4 글자색·유형 구분(기존 엑셀 관례와 정합)

사이트 또는 일정 유형에 따라 글자색을 달리 하여, 인쇄·화면 모두에서 구분이 되어야 한다. 구체 색상 코드는 **코드관리 또는 테마 설정**으로 조정 가능하나, 기본 정의는 다음과 같다.

| 구분 | 표시 예 | 설명 |
|------|---------|------|
| **일반 방문 사이트** | 녹색 계열 | 자동 배치·일반 지정 사이트 |
| **고정 규칙·지정일 사이트** | 적색 계열 | 17.6의 고정 규칙(예: 매월 마지막 주 월요일)에 해당하는 방문 |
| **사무소·본부 등 내부 구분** | 청색 계열 | 사무소 출근·내부 업무 등 코드로 정의된 항목 |

추가 범주가 필요하면 코드로 확장한다.

#### 17.11.5 데이터 일치

- 엑셀 내용은 화면의 **월간 정기점검 일정**과 **동일한 배치 결과**를 반영해야 한다. 화면에서 수정한 뒤 파일을 다시 받으면 **수정 반영본**이어야 한다.
- 선택 기능: 파일 상단 또는 시트에 **연도·작성자·승인 여부** 등 메타정보 행을 넣을 수 있다.

#### 17.11.6 구현 시 참고(비기능)

- 셀 **자동 줄바꿈**, **행 높이·열 너비** 조정로 다줄 사이트명이 잘리지 않게 한다.
- 인쇄 시 가독성을 위해 **가로 방향(A4/A3)** 프리셋을 제공할 수 있다(선택).

## 18. 결론

고객지원 시스템의 성공 여부는 AS 화면보다 먼저, 기준정보의 완성도에 달려 있다. 따라서 본 시스템은 고객기관, 공간, 담당자, 설치자산, 수행관계 정보를 중심으로 설계해야 하며, 이를 토대로 AS 접수·처리·현황을 안정적으로 운영해야 한다.

또한 축적된 데이터는 단순 운영 기록에 머무르지 않고, 타사 납품 자산의 교체시기 파악과 당사 교체영업 제안으로 이어질 수 있어야 한다. **정기 점검 일정 기능**은 이러한 기준정보(기관·위치·자산)를 활용해 현장 운영 효율을 높이는 확장 축이며, **엑셀 월별 캘린더**로 기존 유지보수계획 관례와 맞춘 공유·보관을 지원한다. 본 기획서는 이러한 목적에 맞춘 고객지원 시스템의 기준 모델로 활용한다.
