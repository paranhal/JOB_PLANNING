# 업무일지 프로그램 기획서 (통합형)

**목적:** 제조사·품목·지역별로 나뉘어 있던 업무일지를 **하나의 통합 시스템**으로 새로 설계·구축한다. **향후 지역이 추가되거나 새로운 장비·품목이 추가되어도** 동일한 통합 구조로 수용할 수 있게 확장 가능하게 둔다.

**업무 맥락:** 도서관 자동화 장비·LAS·홈페이지 등 영업 및 현장 기술 지원 — 정기 점검, 담당자, AS·문의 대응 관리. 고객/장비 이력까지 한 곳에서 관리.

**설계 원칙:**  
- **통합 우선** — 처리 이력·고객·장비·일정은 각각 **한 종류의 데이터**로 설계하고, 제조사·품목·지역 등 “구분”은 **확장 가능한 속성**으로 둔다.  
- **확장 가능** — 새 지역·신규 장비·새 품목이 추가돼도 구분(코드)만 추가하면 되도록, 고정 열거가 아닌 **분류 체계**로 설계한다.  
- **기존 시트는 참고·이관용** — 과거 엑셀은 데이터 가져오기/내보내기 시 매핑 참고용으로만 활용한다.

**문서 구성:** §0 통합 목표 · §1 통합 핵심 기능 · §2 통합 데이터·입력 구조 · §3 MVP · §4 확장 · §5 개발 순서 · §6 비기능 · §7 용어 · §8 수용 기준 · §9 우선순위 · §10 가정·제약·리스크 · §11 사용자 시나리오 · §12 웹 운영 전환 시 유의사항 · 부록(기존 시트 참고)

## 0. 통합 목표와 설계 원칙

### 0-1. 왜 통합인가

- 현재: 제조사·품목·지역별로 시트가 나뉘어 있고(나이콤 처리리스트, 원콜 통합, 세종 KLAS, 유지보수현황, 충남/세종 KLAS 등) **지역이 추가되거나 신규 장비가 생기면** 시트·양식이 또 늘어나는 구조.
- 목표: **처리 이력·고객·장비·일정은 각각 하나**로 두고, **제조사·품목·지역은 “구분”으로만 구분**. 새 지역·신규 장비가 생겨도 **구분(분류)만 추가**하면 되도록 확장 가능하게 설계.

### 0-2. 품목·제조사·지역 구조 (현재 기준, 확장 가능)

통합 시 “구분”에 넣을 수 있는 값은 **고정이 아니라 확장 가능**하다. 현재 파악된 구성을 아래에 정리한다.

| 제조사/담당 | 품목/라인 | 비고 |
|-------------|-----------|------|
| **앤로보틱스**(구, NICOM) | **RFID** (자가대출반납기 등) | 기존 나이콤 유지보수·AS |
| | **로봇** | |
| **채움씨앤아이** | **KLAS** — 충남 통합 KLAS | 충남교육청 등 |
| | **KLAS** — 세종 통합 KLAS | 세종시도서관 등 |
| **당사 제작** | **통합 홈페이지** | 충남/제천시립도서관 등 |
| **기타** | 카드프린터 등 | 그 외 품목 |

- **접수 방법**은 별도 속성: **유선, 대면, 메일, 기타, 원콜센터**. 원콜센터는 웹에 접수 페이지·관리 페이지가 있음. 품목·제조사(구분)와 별도로 필터·집계 가능.
- **향후**: 다른 지역(시·도) 통합 KLAS, 새 제조사·신규 장비·새 품목이 추가돼도 **구분(분류) 코드만 추가**하면 되도록 설계. (코드 테이블 또는 관리자가 추가 가능한 분류로 구현.)

### 0-3. 통합 후 구조 (개념)

| 통합 영역 | 설명 | 구분(제조사·품목·지역) 역할 |
|-----------|------|-----------------------------|
| **통합 처리 이력** | AS·유지보수·문의·정기방문·현장방문 등 **모든 건**을 한 목록으로 관리 | 각 건에 “구분” 속성(앤로보틱스 RFID/로봇, 채움 KLAS 충남/세종, 당사 홈페이지, 기타 등). 필터·정렬·내보내기 시 사용. **추가 분류 가능.** |
| **통합 고객·담당자·장비** | **한 마스터**에 고객·담당자·장비·점검주기·유무상 등 보관 | 고객/장비별 “구분”으로 제조사·품목·지역 구분. 처리 이력에서 동일 마스터에서 선택. **새 구분 추가 가능.** |
| **통합 월별 일정** | **한 달력**에 날짜별 방문처·고객·메모 | 구분별 색상/범례는 표시 옵션(선택). |

### 0-4. 기존 시트와의 관계

- **설계 기준은 “통합 구조”**이다. 시트 이름·시트 개수에 맞출 필요 없음.
- 기존 엑셀은 **데이터 이관(가져오기)** 또는 **내보내기로 기존 양식에 맞춰 줄 때**만 참고한다.
- 상세 시트 목록·컬럼은 `엑셀_시트_분석결과.md` 및 본 문서 **부록**에만 정리.

## 1. 통합 핵심 기능

| 구분 | 기능 | 설명 |

## 2. 통합 데이터·입력 구조

### 2-1. 통합 처리 이력 — 입력 항목

**한 종류의 “처리 이력” 레코드.** 구분(라인)에 따라 필수/선택 필드만 다르게 적용.

| 항목 | 필수/선택 | 비고 |
|------|-----------|------|
| **구분** | 필수 | 제조사·품목·지역 등 **확장 가능한 분류**. 예: 앤로보틱스(RFID/로봇), 채움 KLAS(충남/세종), 당사 홈페이지, 기타. 신규 지역·장비 추가 시 코드만 추가. |
| **일시**(또는 일자+처리일자) | 필수 | 접수·처리 기준일 |
| **고객**(또는 도서관명) | 필수 | 통합 고객 마스터와 연결 또는 자유 텍스트 |
| **담당자** | 선택 | 마스터 담당자와 연결 또는 자유 텍스트 |
| **진행 상태** | 필수 | 완료 / 처리완료 / 진행중 등 |
| **접수 방법** | 선택 | **유선**, **대면**, **메일**, **기타**, **원콜센터**. 원콜센터는 웹 접수 페이지·관리 페이지가 있음. 필터·집계 시 사용. |
| **접수 내용** | 선택 | 나이콤형에서 주로 사용 |
| **처리 내용** | 선택 | 나이콤형에서 주로 사용 |
| **분류** | 선택 | 업무 유형: 정기방문·현장방문·메일문의 등 (접수 방법과 별도) |
| **제목** | 선택 | 원콜/세종KLAS형 |
| **내용** | 선택 | 원콜/세종KLAS형 |
| **답변** | 선택 | 원콜/세종KLAS형 |
| **모델명·장비위치** | 선택 | 나이콤형에서 주로 사용 |
| **유/무상** | 선택 | 나이콤형 |
| **비고** | 선택 | 공통 |

→ **통합 모델 하나**로 두고, 화면/유효성만 “구분”에 따라 다르게 줄 수 있음.

## 3. MVP(1차 범위) — 통합 기준

**원칙:** “통합 처리 이력 + 통합 고객·담당자·장비 + 통합 월별 일정”을 한 세트로 동작시키는 것이 MVP. 기존 시트 개수·이름과 무관하게 **통합 구조가 동작하는 것**이 목표.

### 3-1. MVP 포함

| 순위 | 기능 | 내용 |
|------|------|------|
| 1 | **통합 처리 이력** | 한 목록에 모든 구분(제조사·품목·지역) 입력·조회·수정·삭제. 기간·구분·고객·진행 필터. 고객/담당자는 통합 마스터에서 선택. 구분은 확장 가능. |
| 2 | **통합 고객·담당자·장비** | 한 마스터에 고객·담당자·장비·구분·점검주기·유무상 등 CRUD. 구분별 필터. 신규 구분(지역·장비) 추가 가능. |
| 3 | **통합 월별 일정** | 연·월별 달력 뷰, 날짜별 방문처·고객 목록(텍스트)·메모 입력·수정·삭제. |
| 4 | **저장** | 로컬(JSON 파일 또는 로컬 PostgreSQL) 즉시 반영, 재실행 시 유지. 확장 시 서버 PostgreSQL 전환. |
| 5 | **엑셀 내보내기** | 통합 데이터를 필요 시 기존 시트 양식에 맞춰 내보내기(2차에서 상세 매핑). |

### 3-2. MVP에서 제외(2차)

- 고객/장비별 **이력 통합 조회** 화면  
- 통계·예산·AS 요약  
- **엑셀 가져오기**(기존 시트 → 통합 DB 이관)  
- 다음 점검일 자동 계산, 알림  
- 다중 사용자·권한, 서버·클라우드

## 7. 용어 정리

| 용어 | 설명 |
|------|------|
| **통합 처리 이력** | 제조사·품목·지역 구분 없이 한 목록으로 관리하는 업무 기록 전체. 구분은 확장 가능. |
| **통합 고객·담당자·장비** | 구분(제조사·품목·지역)을 속성으로 갖는 단일 마스터. 신규 구분 추가 가능. |
| **구분** | 제조사·품목·지역을 나타내는 **확장 가능한 분류**. 예: 앤로보틱스(RFID/로봇), 채움 KLAS(충남/세종), 당사 홈페이지, 기타(카드프린터 등). 지역·신규 장비 추가 시 값만 추가. |
| **접수 방법** | 건별로 “어떤 경로로 접수되었는지”. **유선, 대면, 메일, 기타, 원콜센터**. 원콜센터는 웹 접수 페이지·관리 페이지 보유. |
| **유/무상, 점검주기** | 유지보수 조건·주기. |

## 12. 웹 운영 전환 시 유의사항

현재 MVP는 **로컬 실행·단일 사용자**를 전제로 한다. 향후 **웹 운영**(브라우저 접속·서버 배포)으로 전환할 때 기획·설계 단계에서 미리 고려할 점을 정리한다.

| 구분 | 유의사항 |
|------|----------|
| **데이터·이관** | 로컬(JSON 또는 로컬 PostgreSQL) → 서버 PostgreSQL 이관 경로·스크립트를 사전 설계. 데이터 설계 문서(`데이터_처리_설계_PostgreSQL확장.md`)와 동일 스키마·필드명을 쓰면 이관이 단순해짐. **등록일시·수정일시** 등 메타 필드 보존 여부를 정책으로 정할 것. |
| **보안·개인정보** | 담당자 연락처·이메일 등이 서버에 올라감. **HTTPS**, 접근 권한(로그인·역할), 개인정보 보관·파기 정책, 최소 수집 원칙을 정해야 함. |
| **다중 사용자** | 웹 전환 시 동시 접속·역할(담당자/관리자)이 생길 수 있음. **동시 편집**(같은 고객/이력 수정), **권한**(조회만/수정/삭제), **감사 로그**(누가 언제 수정했는지)를 기획 단계에서 범위에 넣을 것. |
| **원콜센터 연동** | 원콜센터에 웹 접수·관리 페이지가 있으므로, 업무일지 웹화 시 **원콜센터 ↔ 업무일지** 연동(API·자동 접수 반영·SSO 등) 수요가 생길 수 있음. 경계·역할 분담(어디서 접수하고 어디서 이력 관리하는지)을 미리 정리할 것. |
| **가용성·백업** | 로컬은 “내 PC”에만 의존; 웹은 **서버 장애 = 전원 이용 불가**. 백업·복구 절차, 가능하면 정기 자동 백업과 복원 훈련을 운영 계획에 포함할 것. |
| **오프라인** | 로컬은 네트워크 없이 동작; 웹은 **접속 끊기면 사용 불가**. 현장·이동 중 사용이 많다면 PWA·오프라인 캐시·동기화 전략을 검토할 것. |
| **설계 시 선행 고려** | 로컬 1차라도 **데이터 구조·API 형태**를 나중에 웹 백엔드로 옮기기 쉽게 두면 전환 비용이 줄어듦. (예: 저장소 계층 분리, 엔티티명·필드명 통일.) |

→ **요약:** 웹 전환 시에는 **이관·보안·다중 사용자·원콜 연동·가용성·오프라인**을 기획 단계에서 함께 정하고, 로컬 단계에서도 구조를 웹 전환에 대비해 두는 것이 좋다.

## 요약

- **통합이 핵심:** 제조사·품목·지역(앤로보틱스 RFID/로봇, 채움 KLAS 충남/세종, 당사 홈페이지, 기타 등)을 시트별로 나누지 않고, **통합 처리 이력·통합 고객·담당자·장비·통합 월별 일정** 한 세트로 설계·구현한다. 구분은 **확장 가능한 속성**으로 두어, 향후 지역 추가·신규 장비 추가 시 코드만 추가하면 된다.
- **기존 시트:** 설계의 주된 기준이 아니며, 데이터 이관·내보내기 시 참고용으로만 활용한다.
- **MVP:** 통합 처리 이력 + 통합 고객·담당자·장비 + 통합 월별 일정이 한 흐름으로 동작하고, 구분 추가·필터·로컬 저장(JSON 또는 로컬 PostgreSQL)·내보내기가 가능한 수준. 확장/웹 시 서버 PostgreSQL.
- **확장:** 이력 통합 조회, 통계, 예산, 가져오기, 알림 등은 2차에서 통합 구조를 유지한 채 추가.

이 기획서를 기준으로 **통합 데이터 구조(§2)** 와 **통합 기능(§1)** 부터 구현하면 된다. 기존 엑셀 시트 상세는 `엑셀_시트_분석결과.md` 및 아래 부록 참고.
